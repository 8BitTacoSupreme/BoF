## Component Inventory

New components for Phase 3. Each maps to a decision from `03-CONTEXT.md`.

### 1. Push Live Button (inline in QueryBuilder.jsx)

Location: Below the existing Generate SQL / Validate row, to the right of the validation indicator.

States:
| State | Appearance | Condition |
|-------|------------|-----------|
| Enabled | bg-cyan-600 hover:bg-cyan-500, white text, "Push Live" | validationStatus === 'green' or 'yellow' AND no active job for this session |
| Disabled (red) | bg-gray-700 text-gray-500 cursor-not-allowed, "Push Live" | validationStatus === 'red' or null |
| Disabled (job active) | bg-gray-700 text-gray-500 cursor-not-allowed, "Job Running" | deployment state is Running or Streaming |
| Loading (submitting) | spinner + "Submitting…" | deployment state is Submitting |

Keyboard shortcut: Cmd+Shift+Enter (displayed as ⌘⇧↵ hint next to button — same pattern as ⌘↵ for Generate SQL).

Source: 03-CONTEXT.md D-305, D-308.

### 2. DeploymentStatusPanel (new component, rendered below SqlEditor in QueryBuilder)

Visibility: Renders only after user clicks Push Live. Persists until user closes it or navigates away.

Layout: bg-gray-800 rounded-md border border-gray-700 p-4, full width of main panel, flex-col gap-3.

Sections (top to bottom within panel):
1. Status row: job state pill + output topic name + elapsed time since submit
2. Live messages sub-panel (scrolling table, max-height 240px, overflow-y-auto)
3. Error area (conditional, only on error states)
4. Action row: Stop button (when Running/Streaming), Retry button (when FAILED), Edit SQL link (when submission error)

Close control: small "×" button top-right corner, bg-transparent text-gray-400 hover:text-gray-200, aria-label="Dismiss deployment status".

Source: 03-CONTEXT.md D-306, D-310, D-313.

### 3. Job State Pill

Inline element inside DeploymentStatusPanel status row.

| State | Pill Style | Dot Color |
|-------|------------|-----------|
| Submitting | bg-gray-700 text-gray-300, animated pulse dot | gray-400 (pulsing) |
| Compiling | bg-yellow-900/30 text-yellow-300, animated pulse dot | yellow-400 (pulsing) |
| Starting | bg-yellow-900/30 text-yellow-300, animated pulse dot | yellow-400 (pulsing) |
| Running | bg-green-900/30 text-green-300, steady dot | green-500 (steady) |
| Streaming | bg-cyan-900/30 text-cyan-300, steady dot | cyan-400 (steady) |
| FAILED | bg-red-900/20 text-red-400, no dot | — |
| STOPPED | bg-gray-700 text-gray-400, no dot | — |

Dot: 8px (w-2 h-2) rounded-full, inline-flex items-center gap-2. Pulsing states use `animate-pulse` class.

Source: 03-CONTEXT.md D-306, D-313. Colors consistent with ValidationIndicator.jsx hex values.

### 4. Live Messages Table (inside DeploymentStatusPanel)

Behavior: polls GET /api/topics/:topic/messages?limit=10&since={offset} every 2s while panel is open and state is Running or Streaming.

Layout: scrollable table, max-height 240px, newest message at top, columns derived from output schema fields.

| Element | Style |
|---------|-------|
| Table wrapper | bg-gray-900/50 rounded border border-gray-700/50 overflow-auto max-h-60 |
| th | text-xs text-gray-500 font-mono px-3 py-1.5 text-left border-b border-gray-700 bg-gray-800 sticky top-0 |
| td | text-xs text-gray-300 font-mono px-3 py-1.5 border-b border-gray-700/30 |
| Offset/timestamp column | text-xs text-gray-600 (muted, rightmost column) |
| Row hover | hover:bg-gray-800/50 |

Empty / no-message state (after 30s): Replace table with muted message text (see Copywriting section).
Loading state (initial fetch): Show 3 skeleton rows — bg-gray-700/40 animate-pulse, h-6 per row.

Source: 03-CONTEXT.md D-310, D-311. Inherits table styling from Phase 2 SampleOutput.jsx pattern.

### 5. Derived Topic Badge in Schema Sidebar

Location: Appended to topic name row inside the existing SchemaSidebar component.

Element: `<span>◆ derived</span>` — text-xs text-cyan-300 bg-cyan-900/20 rounded px-1.5 py-0.5 font-mono ml-1.

When shown: topic objects with `isDerived: true` flag (from extended /api/schemas response, per D-309).

Source: 03-CONTEXT.md D-309.

### 6. Stop Job Confirmation (inline, not modal)

Trigger: Clicking "Stop" button while job is Running or Streaming.

Pattern: Inline replacement of the Stop button with a confirmation row — no modal overlay, no full-screen dialog.

```
[Stop this job?]  [Yes, stop]  [Cancel]
```

- "Stop this job?" — text-sm text-gray-300
- "Yes, stop" — text-sm text-red-400 hover:text-red-200 underline cursor-pointer
- "Cancel" — text-sm text-gray-500 hover:text-gray-300 underline cursor-pointer

Auto-dismisses on cancel. No secondary effects beyond halting the Flink statement.

Source: 03-CONTEXT.md D-308. Inline confirmation pattern mirrors Phase 2 error dismiss ("Try again" link pattern).

## Interaction Flows

### Push Live Flow (primary)

1. User reviews generated SQL in Monaco editor. Validation indicator shows green or yellow.
2. Push Live button is enabled. User clicks or presses Cmd+Shift+Enter.
3. Button enters "Submitting…" spinner state immediately (optimistic UI — no network delay displayed to user before feedback begins).
4. DeploymentStatusPanel opens below the SQL editor with state pill = "Submitting".
5. State pill transitions: Submitting → Compiling → Starting → Running → Streaming as backend polling reflects SQL Gateway lifecycle.
6. Once Running: Live Messages Table begins polling. First rows appear within 2–4s.
7. Schema Sidebar auto-refreshes (re-calls /api/schemas). Derived topic appears with ◆ badge.
8. Push Live button becomes "Stop" button (disabled while Submitting/Compiling/Starting; enabled at Running).

### Stop Flow

1. User clicks "Stop" (only available when job state is Running or Streaming).
2. Inline confirmation row replaces the Stop button.
3. User clicks "Yes, stop" — backend halts statement, state pill transitions to STOPPED.
4. "Yes, stop" triggers DELETE /api/jobs/:id (or equivalent halt endpoint).
5. Live Messages Table stops polling. Final messages remain visible.
6. Stop button is replaced with a static "Stopped" label. No restart option in this phase.

### Error Recovery Flows

Submission error (SQL Gateway 400):
1. State pill shows FAILED. Error area in panel shows gateway error text.
2. "Edit SQL" link returns focus to Monaco editor cursor at line 1.
3. DeploymentStatusPanel remains open so user can see what failed.

Runtime error (job transitions to FAILED):
1. State pill shows FAILED. Error area shows Flink exception summary (first 200 chars).
2. "Retry" button re-submits the same SQL. No changes to editor required unless user edits first.

Connectivity error (SQL Gateway unreachable):
1. State pill shows error text with countdown: "Retrying in 4s…".
2. Auto-retry fires at 0. If fails again: show "Gateway unreachable — check Docker Compose" with manual "Retry" button.
3. No automatic infinite retry loop — max 3 auto-retries then manual.

No-messages-after-30s:
1. Live Messages Table replaced by no-message state copy.
2. "[Check Job Status]" link is a text button that fetches GET /api/jobs/:id and surface the raw state — no navigation, inline display only.

Source: 03-CONTEXT.md D-305 through D-316.

## Interaction States Summary

All interactive elements must handle these states explicitly:

| Component | States |
|-----------|--------|
| Push Live button | enabled / disabled-red / disabled-job-active / loading-submitting |
| Stop button | enabled / loading-stopping / confirmation-shown |
| DeploymentStatusPanel | hidden / submitting / compiling / starting / running / streaming / failed / stopped |
| Live Messages Table | loading-skeleton / live-data / empty-30s / error |
| Schema Sidebar topic row | normal / expanded / derived-badge |
| Job state pill | all 7 states listed in Component Inventory |

Polling behavior:
- Live Messages: every 2s, only when DeploymentStatusPanel is visible AND state is Running or Streaming. Stop on panel close or state change to FAILED/STOPPED.
- Job status: every 5s, only when DeploymentStatusPanel is visible AND state is not terminal (FAILED, STOPPED). Uses GET /api/jobs/:id.
- Schema Sidebar refresh: one-shot re-call of /api/schemas triggered at state transition Running. No continuous polling.

## Design Continuity Notes

Phase 3 extends Phase 2 in-place. No new views, no navigation redesign. Specific continuity rules:

1. QueryBuilder.jsx remains the single pane of glass. DeploymentStatusPanel renders inside it, below the SQL editor section.
2. SchemaSidebar receives one new visual element (◆ derived badge) but its structure, colors, and expand/collapse behavior are unchanged.
3. All existing Tailwind classes carry forward verbatim. No new color tokens, no new font sizes, no new spacing values.
4. The Monaco editor (SqlEditor.jsx) is unchanged — same vs-dark theme, same 14px font, same options. The Push Live button appears outside the editor, not inside it.
5. Error display pattern (`border border-red-700 bg-red-900/20 rounded-md p-3`) from Phase 2 QueryBuilder.jsx is reused verbatim for submission errors in DeploymentStatusPanel.
6. The inline "Try again" link pattern from Phase 2 error display is reused verbatim for "Edit SQL" and "Retry" links.
